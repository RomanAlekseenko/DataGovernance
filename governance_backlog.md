# Governance Backlog

## Назначение
Этот файл используется для фиксации кросс-доменных задач по развитию `Data Catalog` и `Data Governance`, которые не относятся только к одному домену.

Если будущие действия относятся к конкретному домену, они должны фиксироваться в соответствующем `to_do.md` этого домена.

## Правила ведения
* Каждый пункт должен быть сформулирован как конкретное следующее действие.
* Если задача относится к домену, в пункте должна быть ссылка на доменный `to_do.md`.
* Если задача блокируется отсутствием решения, это должно быть отмечено явно.
* Закрытые пункты можно переносить в историю изменений или удалять после фиксации результата в основном артефакте.

## Кросс-доменные задачи
- [ ] Создать единый шаблон для описания источников данных и витрин на базе `_templates/dataset_example.md`.
- [ ] Создать единый каталог glossary-терминов и определить правила размещения файлов в `Product/glossary`.
- [ ] Зафиксировать naming conventions для доменов, метрик, датасетов и словаря.
- [ ] Определить единый подход к versioning определений и маркировке `breaking change`.
- [ ] Определить минимальный стандарт quality checks для метрик и источников.
- [ ] Зафиксировать стандарт ведения `to_do.md` для доменов и кросс-доменных governance-задач.
- [ ] Расширить glossary стартовым набором терминов `заказ`, `выручка`, `платящий пользователь`, `конверсия`, `возврат`.
- [ ] Определить, нужен ли отдельный каталог dashboard inventory помимо доменных `dashboard_map.md` и `dashboard_requirements.md`.
- [ ] Зафиксировать кросс-доменный ownership для glossary: кто является business owner и кто выполняет роль data steward для терминов `пользователь`, `сессия`, `транзакция`, `authoritative source`.
- [ ] Определить, нужен ли единый cross-domain стандарт identity resolution и в каких контурах допустимы `device_id`, `user_id` и `account_id` как primary identifier.
- [ ] Определить единый cross-domain стандарт sessionization, включая правило границы сессии и трактовку расхождений между client-side и server-side сессиями.
- [ ] Зафиксировать различие между терминами `заказ` и `транзакция` и определить, где каждый из них должен использоваться как базовая сущность.
- [ ] Зафиксировать единый стандарт трактовки `gross revenue`, `refund-adjusted revenue` и связанных возвратных сценариев.
- [ ] Определить, нужен ли статус `provisional source` до окончательного подтверждения authoritative source, и какие минимальные атрибуты обязательны для признания источника authoritative.
